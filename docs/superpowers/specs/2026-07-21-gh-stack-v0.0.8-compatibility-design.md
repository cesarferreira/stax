# gh-stack v0.0.8 compatibility design

## Goal

Update Stax's native GitHub Stack integration for `github/gh-stack` v0.0.8 while preserving compatibility with older extension versions. The work will ship as three stacked pull requests so authentication behavior, remote unlinking, and append-only update handling can be reviewed independently.

## Pull request stack

### 1. Support v0.0.8 authentication and doctor diagnostics

The first pull request raises the recommended gh-stack version from v0.0.6 to v0.0.8. `st doctor` will always report the detected extension version. Versions below v0.0.8 will be labeled out of date, included in the `--fix` repair plan, and paired with the canonical `gh extension upgrade stack` command. An unparseable development version will be shown as unknown without being treated as an automatic repair target.

gh-stack v0.0.8 uses the public Stacks REST API and accepts the normal GitHub CLI authentication sources. Stax will therefore preserve `GH_TOKEN` and `GITHUB_TOKEN` for v0.0.8 and newer. For v0.0.6 and v0.0.7, Stax will retain the existing behavior of removing those variables so the extension can fall back to an OAuth-authenticated account. The legacy `AuthTokenUnsupported` outcome remains supported because older installations can still return it.

The repository feature cache remains unchanged. Although the REST endpoint is public, the Stacked PRs feature is still gated per repository.

### 2. Unstack a remote native stack by number

The second pull request adds an optional positive stack-number argument:

```text
st stack unlink [<stack-number>]
```

When a number is supplied, Stax delegates to `gh stack unstack <stack-number>`, which can remove a remote stack without gh-stack local tracking. With no argument, the command keeps its current behavior and targets the locally active gh-stack stack.

Zero and malformed values are rejected by CLI parsing. If argument-free unlink reports that the current branch is not locally tracked, the diagnostic will recommend `st stack unlink <stack-number>` before falling back to the GitHub UI.

### 3. Handle append-only updates and expose stack numbers

The third pull request models v0.0.8's append-only update failures separately from fork conflicts. Messages stating that an update would remove an existing PR or that new PRs must be appended to the top will produce a dedicated outcome. The user-facing diagnostic will explain that the remote stack must be unstacked and linked again, using `st stack unlink <stack-number>` when the number is available.

Successful `gh stack link` output will be inspected for a `stack #<number>` label. When present, Stax will print the number in submit and explicit-link success messages. A successful output without a recognizable number remains successful and keeps the generic message; parsing output must never turn a successful link into a failure.

The existing proactive local-fork check remains in place. The old `would remove ... from the stack` phrase is removed from fork classification because it now describes the append-only contract. The separate `stack contents have changed` conflict classification remains for concurrent or incompatible remote changes.

## Interfaces and data flow

Version detection continues to use `gh extension list`. The parsed version is represented once and reused by doctor reporting and command construction. Command construction chooses whether to remove authentication override variables based on that version.

`link_stack` and `unlink_stack` continue to return typed outcomes rather than exposing raw process output to command modules. Successful link outcomes gain an optional stack number, and append-only failures gain their own variant. CLI commands translate those outcomes into concise guidance and decide whether a failure is fatal (`st stack link`) or non-blocking (`st submit` automatic linking).

No persistent stack-number metadata is introduced. Numbers are displayed when the extension returns them, and users can supply them explicitly to unlink. Automatic persistence or remote stack import can be added later without changing this command contract.

## Error handling

- Missing or outdated extensions retain the existing setup guidance.
- Older extensions retain OAuth fallback and PAT-rejection handling.
- Repository feature-disabled responses remain cached and non-blocking during submit.
- A numbered unstack failure is reported with the underlying gh-stack message.
- Append-only divergence is not cached as feature-disabled and is not mislabeled as a local fork.
- Unknown or changed success-output formatting only suppresses the optional number; it does not fail the operation.

## Verification

Each pull request includes targeted integration coverage for its behavior.

The first pull request covers v0.0.8 token preservation, legacy token stripping, doctor output for current/outdated/unknown versions, and `doctor --fix` upgrade invocation. The second covers numbered unlink, argument-free unlink, and invalid numbers. The third covers number extraction, success without a number, removal/reorder diagnostics, retained fork classification, and generic failures.

During implementation, `make lint-fast` is used for quick feedback. Before submission, the full stack is checked with `make lint` and `make test`, following the repository policies. Documentation changes include `README.md`, the relevant command and integration pages under `docs/`, and `skills.md`.

## Out of scope

- Importing a GitHub native stack into Stax metadata.
- Wrapping gh-stack's interactive checkout picker or sync workflow.
- Persisting stack numbers in Git or Stax metadata.
- Changing Stax branch naming based on gh-stack's removed prefix options.
