# gh-stack v0.0.8 Compatibility Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship three stacked pull requests that update native GitHub Stack authentication and doctor reporting, add remote unlink by stack number, and handle v0.0.8 append-only updates while displaying stack numbers.

**Architecture:** Keep process execution and output classification inside `src/github/gh_stack.rs`, with command modules translating typed outcomes into user-facing behavior. Extend the existing CLI without persistent metadata: version detection controls legacy token stripping, unlink accepts an optional number, and successful link output carries an optional parsed stack number.

**Tech Stack:** Rust 2024, clap, `std::process::Command`, existing subprocess integration-test harness, Markdown documentation, Stax stacked branches.

## Global Constraints

- Recommend exactly gh-stack v0.0.8 or newer while preserving functional compatibility with older link-capable versions.
- Preserve `GH_TOKEN` and `GITHUB_TOKEN` for v0.0.8+; strip them only for known older versions.
- Keep the repository feature-enabled cache because Stacked PR access remains repository-gated.
- Do not persist stack numbers or import remote stacks into Stax metadata.
- Use `make lint-fast` during implementation, `make lint` before submission, and `make test` for the full suite.
- Update `README.md`, relevant `docs/` pages, and `skills.md` with every user-visible behavior change.

Treat successful unlink as `Linked { .. }` and ignore its optional number.
